# Rocket Elevators — AND-2

**Purpose:** Operations dashboard and data pipeline analyzing Ontario elevator fleet status using AI-assisted workflows.

## Tech Stack

- **Backend:** Python 3.10+, Flask, Pandas
- **Frontend:** HTMX 2.0.4, Tailwind CSS, Jinja2 — no custom JavaScript
- **Analysis:** Jupyter, scikit-learn, NLTK
- **Database:** PostgreSQL 16 (via Docker) — Go API reads all operational data from here

## Directory Structure

- `/data` — source datasets (read-only, never modify in-place)
- `/platform` — Flask app, `elevator_fleet.csv`, HTML templates
- `/intelligence` — Jupyter notebooks (ETL, NLP, exploratory analysis)
- `/docs` — `dashboard_spec.md` is the single source of truth for the dashboard

## Commands

```bash
# Start full stack (PostgreSQL + Go API)
docker-compose up

# Populate PostgreSQL from source datasets (idempotent — safe to re-run)
py -3 intelligence/etl_to_database.py

# Populate risk_explanation for HIGH-risk elevators (requires DB running)
py -3 intelligence/generate_explanations.py

# Regenerate elevator_fleet.csv from /data sources
py -3 platform/prepare_data.py

# Regenerate data/predictions.csv — CLI mirror of generate_predictions.ipynb
py -3 intelligence/generate_predictions.py

# Serve dashboard at http://localhost:5000 (requires Go API running)
py -3 platform/server.py

# Serve Go API at http://localhost:8080 (requires PostgreSQL running)
# Port is 8080, not 8081 — 8081 is blocked by the company firewall. Do not change this port.
go run ./platform/api
```

## Branching Strategy

```
main ← dev ← feat/<card-id>-<short-description>
```

- **`main`** — stable, production-ready. Never commit directly. Receives merges from `dev` only at end of sprint via PR.
- **`dev`** — integration branch. All feature PRs target this branch. Never commit directly.
- **`feat/...`** — short-lived feature branches (1-2 days max). Always branch from `dev`. One task per branch, one branch per PR.

Branch naming: `feat/chat-1-flask-endpoint`, `feat/infra-1-deploy`, etc.
PR rule: at least 1 reviewer required before merging into `dev`.
Merged branches must be preserved — do not delete after merge.

## Conventions

- **Spec-driven:** `platform/index.html` may only reflect content explicitly defined in `docs/dashboard_spec.md`. Update the spec first, then the HTML — never the reverse.
- `elevator_fleet.csv` is written by `prepare_data.py` and read by the Go API via ETL — `server.py` does not read it.
- `server.py` reads `data/merged_elevator_data.csv` and `data/incident.json` only for incident/alteration counts in the elevator detail panel. All summary cards and fleet table data come from the Go API.
- `data/predictions.csv` is generated by `generate_predictions.py`. It is loaded into PostgreSQL by `etl_to_database.py`; the Go API reads predictions from the database, not from the CSV directly.
- `risk_explanation` in the `predictions` table is populated by `generate_explanations.py` after ETL. Rows without a generated explanation have `risk_explanation = NULL`.
- Data pipeline filters to ACTIVE + BY REQUEST licenses only. Join key across all datasets: `ElevatingDevicesNumber`.
- Inspections are one-to-many — always keep only the most recent record per elevator.
